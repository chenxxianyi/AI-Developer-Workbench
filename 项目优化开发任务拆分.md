# AI Developer Workbench 项目优化开发任务拆分

> **执行说明：** 本文由 `PROJECT_OPTIMIZATION_PLAN.md` 拆分而来，用作研发排期、任务领取、联调和验收的主任务清单。每张任务卡应独立开发、独立测试、独立提交；只有通过对应里程碑门禁后，才进入下一阶段。

**目标：** 将当前五工具工作台升级为面向 AI Coding 项目的质量控制台，形成“工具运行 → 报告沉淀 → 行动计划 → AI 修复 Prompt → 复查对比 → 项目趋势”的完整闭环。

**适用范围：** 个人本地、单用户使用；所有任务优先降低个人维护成本和日常使用复杂度。

**总体架构：** 保持现有 Vue 3 + Pinia 前端、Gin + GORM 后端和 MySQL 数据库的分层结构。先完成 P0 产品闭环和部署地基，再扩展统一行动项、项目档案、异步任务与 AI 观测；新增能力优先服务个人本地使用，通过小接口和小组件接入，避免一次性重写五个工具。

**技术栈：** Vue 3、TypeScript、Pinia、Vue Router、Vite、Vitest、Gin、GORM、MySQL 8、Docker Compose、Nginx。

## 1. 文档边界与执行规则

### 1.1 与原方案的关系

- `PROJECT_OPTIMIZATION_PLAN.md` 负责说明产品方向、优先级和目标。
- 本文负责把方向拆成可领取、可开发、可测试、可验收的任务卡。
- 本文不改变原方案的优先级：先 P0 闭环，再 P1 工作流和项目化，最后按个人实际需要选择 P2 专业配置与外部集成。
- 五个工具专项增强是独立工作流，在统一行动项协议稳定后并行推进。

### 1.2 任务状态约定

- `[ ]`：未开始。
- `[~]`：进行中；开始时在任务标题后补负责人和分支名。
- `[x]`：代码、自动化测试、文档和验收均完成。
- 被其他任务阻塞时，在任务标题下记录阻塞任务编号，不用“代码已完成但未联调”冒充完成。

### 1.3 优先级与复杂度

| 标识 | 含义 |
| --- | --- |
| P0 | 当前 MVP 可用性或数据一致性的发布阻断项 |
| P1 | 形成持续质量工作流所需的核心能力 |
| P2 | 个人高级配置与外部工具集成能力 |
| S | 单一模块的小改动，通常不跨前后端 |
| M | 跨 2～3 个模块，需要联调 |
| L | 涉及数据模型、API 和页面的完整纵向切片 |
| XL | 需要进一步拆成多张子任务卡后实施 |

### 1.4 通用完成定义

每张任务卡完成时必须同时满足：

- [ ] 业务行为符合任务卡验收标准。
- [ ] 新增或修改的后端逻辑有 Go 测试；前端交互有 Vitest 组件或工具函数测试。
- [ ] `cd backend; go test ./...` 通过。
- [ ] `cd frontend; npm exec vitest run` 通过。
- [ ] `cd frontend; npm run build` 通过。
- [ ] API、环境变量或部署方式变化已同步到 README、`.env.example` 或接口说明。
- [ ] 不提交密钥、构建产物、上传文件、临时文件和本机配置。
- [ ] 一个任务使用一个可回滚提交，提交信息带任务编号，例如 `feat(P0-03): implement reports list`。

### 1.5 已核实的仓库基线

- `frontend/src/pages/ReportsPage.vue` 和 `frontend/src/pages/ReportDetailPage.vue` 目前仍是占位页。
- 报告列表、详情、删除、Markdown 导出和生成文件下载的后端接口已经存在。
- 报告列表已支持工具筛选、时间/评分排序和分页，但还没有状态筛选。
- `backend/internal/config/config.go` 目前没有 `MockMode` 字段，并且强制要求 `AI_API_KEY`。
- `docker-compose.yml` 已声明前端镜像，但 `frontend/Dockerfile` 尚不存在。
- `backend/internal/service/report_service.go` 当前先更新成功状态，再单独写生成文件，写入失败只记录日志。
- 前端已有 Vitest 依赖和测试文件，但 `frontend/package.json` 尚未提供统一 `test` 脚本。
- `backend/migrations/0001_init.sql` 是基线 SQL；运行时 `backend/internal/database/database.go` 使用 GORM `AutoMigrate`，新增表时必须明确迁移执行策略。
